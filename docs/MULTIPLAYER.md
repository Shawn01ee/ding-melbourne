# Phase 5 — 멀티플레이 구현 계획

## 권장 순서

처음부터 자유 입장 대규모 실시간 방을 만들지 않는다. 현재 게임은 클라이언트에서 결정적으로 진행되므로 아래 순서가 가장 안전하다.

1. **Ghost Challenge** — 친구의 완료 기록을 재생하는 비동기 대결
2. **Private 1v1** — 초대 코드로 들어가는 두 명의 실시간 레이스
3. **Quick Match** — 공개 매칭, 시즌·신고·제재 포함

Ghost Challenge가 1단계인 이유는 연결 끊김·동시 시작·Presence 비용 없이 진행 표시, 공정성 검증, 결과 비교 UI를 먼저 검증할 수 있기 때문이다.

## 공통 경기 규칙

- 양쪽은 동일한 `route`, `direction`, `mode`, `difficulty`, `startStopIndex`를 사용한다.
- 서버가 `matchId`, 설정, 시작 시각, 짧은 유효기간의 런 토큰을 발급한다.
- 클라이언트는 정답 원문이나 키 입력 문자열을 상대에게 보내지 않는다.
- 네트워크에는 최대 초당 5회 `stopIndex`, 현재 정류장 진행률, combo, errors, elapsed만 보낸다.
- 화면의 상대 트램은 수신 진행률을 보간하되 결과 판정에는 사용하지 않는다.
- 최종 순위는 서버가 접수한 검증 결과로만 결정한다.

## 기술 구조

현재 선택한 Supabase를 그대로 사용한다.

- **Postgres/RPC**: 방 생성, 참가, 준비, 시작 시각, 최종 결과, 재접속 상태
- **Realtime Presence**: 접속/이탈/준비 상태
- **Realtime Broadcast**: 수명이 짧은 진행률 이벤트
- **Edge Function 또는 검증 RPC**: 런 토큰 발급, 결과 제출, 이벤트 다이제스트 검사
- **Vercel/Vite 클라이언트**: 로컬 게임 상태는 현행 reducer가 계속 소유

실시간 이벤트를 DB 행으로 매 프레임 저장하지 않는다. 영구 저장은 경기 설정과 최종 결과만 대상으로 한다.

## 데이터 초안

### `matches`

- `id`, `invite_code`, `status`
- `route_short`, `direction_id`, `mode`, `difficulty`, `start_stop_index`
- `host_id`, `starts_at`, `finished_at`, `created_at`

### `match_players`

- `match_id`, `user_id`, `slot`
- `ready_at`, `connected_at`, `disconnected_at`
- `finish_ms`, `stops`, `accuracy`, `wpm`, `errors`, `result_status`

### `ghost_runs`

- 검증 완료된 경기 설정과 최종 지표
- 100–200ms 간격으로 양자화한 누적 진행률 배열
- 원문 키 입력은 저장하지 않음

RLS는 경기 참가자만 비공개 방을 조회할 수 있게 하고, 초대 코드 참가와 결과 제출은 직접 INSERT가 아닌 RPC로 제한한다.

## 단계별 작업

### M0 — 계정/리더보드 안정화

- Supabase 운영 환경 연결
- 로그인, 표시 이름, 삭제 흐름 E2E
- 점수 제출 중복 방지용 `run_id`와 unique constraint
- 표시 이름 필터·신고·관리자 숨김

완료 기준: 로그인/로그아웃/삭제와 100회 연속 점수 제출 테스트에서 중복·권한 오류가 없다.

### M1 — Ghost Challenge

- 완료된 런의 진행률 샘플 생성
- 결과 화면의 Challenge 링크
- 게임 지도에 반투명 상대 트램과 앞/뒤 거리 표시
- 링크 만료 및 비공개 전환

완료 기준: 서로 다른 기기에서 같은 링크를 열어 동일 설정으로 시작하고, 네트워크가 없어도 내려받은 ghost 재생이 끝까지 결정적이다.

### M2 — Private 1v1

- 6자리 초대 코드, 방장/참가자 2석
- Ready → 서버 기준 3초 카운트다운
- Presence와 5Hz progress broadcast
- 10초 재접속 유예, 기권/무효 처리
- 결과 합의가 아닌 서버 제출 시각과 검증 지표로 승패 판정

완료 기준: 데스크톱↔모바일, 300ms 지연, 5% 패킷 손실, 한 차례 재접속에서도 양쪽 최종 승패가 같다.

### M3 — Quick Match

- 지역·모드별 대기열과 매칭 제한 시간
- 시즌, 실력 지표, 차단/신고
- 관전자 없이 1v1부터 운영
- 비정상 WPM·진행률·제출 빈도 탐지와 관리자 도구

완료 기준: 공개 매칭에 필요한 운영·신고·삭제 절차와 장애 시 롤백 스위치가 준비되어 있다.

## 공정성 설계

- 서버 발급 런 토큰에 경기 설정, 사용자, 시작 시각, 만료 시각을 묶는다.
- 입력 이벤트는 문자 자체 대신 정답/오답 플래그, 상대 시간, 누적 문자 수로 다이제스트한다.
- 서버는 최소 이동 시간, stop 순서, 누적 문자 수, WPM 상한, 이벤트 단조성을 검증한다.
- 탭 숨김·백그라운드 전환은 기존 자동 일시정지 대신 멀티플레이 규칙에 맞춰 기권 또는 계속 흐르는 시계로 명확히 정의한다.
- 클라이언트가 보낸 상대 진행률은 연출용일 뿐 승패 판정 자료가 아니다.

## 먼저 결정할 것

1. 첫 공개 버전을 Ghost Challenge로 할지 Private 1v1까지 포함할지
2. 비회원은 관전/초대 참가가 가능한지, 또는 모든 멀티플레이에 로그인을 요구할지
3. 모바일에서 앱이 백그라운드로 가면 즉시 기권인지 10초 재접속 유예인지
4. 친구 링크 보존 기간(권장 7일)과 ghost 기록 공개 범위
5. 지역 서버 지연보다 플레이어 풀을 우선할지 여부
